## 2. Найденные медленные запросы (до оптимизации)

### Запрос №1. Топ-10 выполненных заказов
```
SELECT id, user_id, status, total_amount, created_at
FROM orders
WHERE status = 'completed' 
ORDER BY total_amount DESC
LIMIT 10;
```

План: последовательный скан orders (100к строк) + фильтр

Execution Time: 17 мс

Причина: нет индекса, используется сортировка

### Запрос №2. Оплаченные заказы за январь-май 2024
```
EXPLAIN (ANALYZE, BUFFERS)
SELECT id, user_id, status, total_amount, created_at
FROM orders
WHERE status = 'paid'
    AND created_at BETWEEN '2024-01-01' AND '2024-06-01';
```

План: последовательный скан по orders + фильтр

Execution Time: 11.6 мс

Причина: нет индекса, значительная часть строк отбрасывается по фильтру

### Запрос №3. Сколько товаров и на какую сумму купил пользователь (топ-10)
```
SELECT 
    o.user_id,
    COUNT(DISTINCT o.id) AS orders_count,
    SUM(oi.quantity) AS total_items,
    SUM(oi.quantity * oi.price) AS total_spent
FROM orders o
JOIN order_items oi ON oi.order_id = o.id
WHERE o.status = 'paid'
GROUP BY o.user_id
ORDER BY total_spent DESC
LIMIT 10;
```

План: последовательные сканы orders и order_items (400к) + сортировки и hash join

Execution Time: 448 мс

Причина: join больших таблиц с сортировкой и агрегацией

### Запрос №4. 
```
SELECT 
    o.id,
    o.user_id,
    o.total_amount,
    o.created_at,
    (
        SELECT osh.changed_at
        FROM order_status_history osh
        WHERE osh.order_id = o.id 
          AND osh.status = 'paid'
        ORDER BY osh.changed_at DESC
        LIMIT 1
    ) AS paid_at,
    (
        SELECT COUNT(*)
        FROM order_items oi
        WHERE oi.order_id = o.id
    ) AS items_count

FROM orders o
WHERE o.status = 'shipped'
  AND EXTRACT(MONTH FROM o.created_at) IN (1,2,3,9,10,11,12)
ORDER BY o.created_at;
```

План: последовательные сканы + подзапрос для каждой строки

Execution Time: 176198 мс (2 минуты 56 секунд)

Причина: подзапросы для каждой строки к таблицам `order_items` и `order_status_history` с многократным сканированием этих таблиц


## 3. Добавленные индексы и обоснование типа

### Индекс №1
```
CREATE INDEX idx_orders_created_at ON orders (created_at);
```
- Какой запрос ускоряет: Q2
- Почему выбран тип: B-Tree стандартный, эффективно поддерживает операции диапазона и сортировку

### Индекс №2
```
CREATE INDEX idx_orders_status_created_at ON orders (status, created_at);
```
- Какой запрос ускоряет: Q2
- Почему выбран тип: B-Tree оптимален для составных условий равенства и диапазона

### Индекс №3
```
CREATE INDEX idx_order_items_order_id ON order_items (order_id);
```
- Какой запрос ускоряет: Q4
- Почему выбран тип: B-Tree также эффективен для джоинов, так как позволяет быстро находить строки по ключу `order_id`

## 5. Партиционирование `orders` по дате
### 5.1 Выбранная стратегия
Выбрана стратегия партиционирования по диапазону (RANGE) с разбиением по кварталам.

### 5.2 Реализация
Партиционирование реализовано через создание новой таблицы `orders_new` с ключом партиционирования `created_at` по кварталам 2024-2026 годов. Добавлена DEFAULT-партиция для данных вне диапазона. 
Выполнен перенос данных и созданы аналогичные индексы по партиционированной таблице.

### 5.3 Проверка эффекта
Написан тестовый запрос к исходной таблице и к партиционированной.

До:
- `Bitmap Index Scan` и `Bitmap Heap Scan` по всей таблице
- Execution Time: 10.5 ms

После:
- `Append` только по нужным партициям
- `Seq Scan` по двум партициям
- Execution Time: 6.9 ms

## 7. Что удалось исправить
- Для Q2 устранены полные сканирования с помощью составного индекса `(status, changed_at)`
- С помощью индексов Q4 подзапросы выполняются через `Index Only Scan` почти в 3 тысячи раз быстрее
- Партиционирование решает проблему избыточного чтения данных при запросах по диапазону дат

## 8. Что не удалось исправить только индексами
- В Q1 и Q3 индекс не используется
Причина: выбирается большая часть таблицы и PostgreSQL считает полное сканирование дешевле

- В Q3 индексы не помогли, так как необходимо обработать большой объём данных и GROUP BY с ORDER BY требуют сортировки и агрегации

- Q4 все ещё относительно медленный после оптимизации (60мс) ввиду подзапросов для каждой строки и функции EXTRACT(MONTH), которая не позволяет эффективно использовать индекс и партиции

## 9. Выводы
- С анализом плана запроса EXPLAIN ANALYZE удобно искать узкие места и оптимизировать запрос
- Индексы наиболее эффективны для селективных запросов, где можно значительно сократить объём данных
- Партиционирование помогает снизить количество и объём чтений данных
- Индексы и партиции не являются универсальным решением, наибольший эффект оптимизации достигается при устранении неэффективных конструкций
