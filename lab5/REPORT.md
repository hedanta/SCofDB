## 1. Реализация Redis-кэша

Ключи кэша:
- Каталог товаров `catalog:v1` - агрегированные данные по всем товарам из таблицы `order_items` - название, количество заказов, проданное количество и средняя цена
- Карточка заказа `order_card:v1:{order_id}` - данные о заказе, пользователе и товарах

`TTL`: 600 секунд (10 минут)

Логика работы:
1. При использовании параметра `use_cache=true` проверяется наличие данных в Redis
2. При cache hit возвращаются кэшированные данные. При cache miss выполняется запрос к БД
3. При `use_cache=true` полученные данные сохраняются в Redis с заданным TTL

## 3. Починка через событийную инвалидацию

Для корректной работы инвалидация кэша была реализована с помощью событий `application/cache_events.py`.
- Событие создаётся в момент изменения заказа - вызывается метод `publish_order_updated`
- `_handle_order_updated` получает уведомление о событии и приступает к инвалидации ключей
- Инвалидируются `order_card_key(order_id)` и `catalog_key()` - удаляются из Redis

Тест: `test_cache_event_invalidation.py`

Запуск: `pytest backend/app/tests/test_cache_event_invalidation.py -v -s`

Результат: кэш инвалидируется (`cached_after is None`), информация обновляется, из кэша возвращаются актуальные данные.


## 4. Rate limiting endpoint оплаты через Redis

Правила:
- По умолчанию ограничения 5 запросов за 10 секунд
- Применяется только к endpoint оплаты: `/api/orders/{order_id}/pay` и `/api/payments/retry-demo`
- Ключ лимита генерируется на основе IP клиента
- При превышении допустимого числа запросов возвращается `429 Too Many Requests`
- Клиенту возвращаюются заголовки:
- - `X-RateLimit-Limit`
- - `X-RateLimit-Remaining`
- - `X-RateLimit-Window`

Тест: `test_payment_rate_limit_redis.py`

Запуск: `pytest backend/app/tests/test_payment_rate_limit_redis.py -v -s`

Результат: 5 запросов на оплату 5 различных заказов прошли успешно, шестой вернул `429 Too Many Requests`. Повторный запрос спустя 10 секунд выполняется успешно.

## 6. Выводы
- Кэш даёт наибольший выигрыш при повторяющихся запросах к одним и тем же данным. Если данные часто меняются или запросы уникальны, то кэш не даст прироста метрик
- При изменении данных нужно явно инвалидировать кэш, чтобы показывать актуальную информацию
- Rate limiting с подходящими ограничениями защищает от перегрузки и дублирования
- Важно подбирать TTL кэша исходя из ожидаемой нагрузки и требований к свежести данных
